# Awesome Seedance — 主编排层

> 作者：Takis/安梓豪 | 仓库：[agent-skill-sync](https://github.com/oijhl852/agent-skill-sync)

将剧本、台词或故事转成 Seedance 2.0 视频提示词，并可选注入到 Seedance 网页面板。

## 子技能速查

### 📝 内容创作

| 调用 | 职责 | 触发场景 |
|------|------|---------|
| `sd-read-script` | 解析剧本格式、提取角色/场景/台词/动作 | 用户提供分场剧本、小说文本 |
| `sd-segment-split` | 按戏剧节拍拆15秒段、反模式检查 | 已有剧本需分段时 |
| `sd-dialogue` | 对话戏分镜（铁律0-3、镜头分配） | 以对白为主的场景 |
| `sd-action` | 动作/打斗/蒙太奇分镜 | 以动作、战斗为主的场景 |
| `sd-story-adapt` | 故事→剧本改编（三幕式、短剧公式） | 用户说"帮我写个剧本" |
| `sd-asset-guide` | 人物/场景/道具四视图出图提示词 | 需要生成素材图时 |

### 🎯 提示词

| 调用 | 职责 | 触发场景 |
|------|------|---------|
| `sd-prompt` | 组装11个技术标签 + 资产引用/时间轴提示词 | 分镜完成后产出最终提示词 |
| `sd-prompt-library` | 精选案例库与两套本地参考框架 | 用户想参考别人怎么写的 |
| `sd-community` | 4776条本地完整语料 + 搜索与评分 | 需要大量灵感或好坏案例对比 |

### ✅ 质量

| 调用 | 职责 |
|------|------|
| `sd-quality` | 保真/密度/约束/衔接检查 |

### 🚀 面板注入

| 调用 | 职责 | 触发场景 |
|------|------|---------|
| `sd-panel` | Chrome 连接、面板元素定位、工具可用性检测 | "注入到 Seedance 面板" |
| `sd-chip` | @mention 触发、chip 数据提取、HTML 构建 | 需要引用资产时 |
| `sd-inject` | 凭证提取、segment/list、segment/update API | 注入提示词到面板 |

## 执行流程

```
输入（剧本/台词/故事/场景描述）
  │
  ├─ 分场剧本/台词 ──→ sd-read-script ──→ sd-segment-split ──→ sd-dialogue|sd-action ──→ sd-prompt
  ├─ 单个场景/镜头 ──→ sd-prompt
  ├─ 故事/大纲 ──────→ sd-story-adapt ──→ sd-read-script ──→ sd-segment-split ──→ ...
  ├─ 灵感/案例参考 ───→ sd-community|sd-prompt-library ──→ sd-prompt ──→ sd-quality
  ├─ 已有提示词优化 ──→ sd-quality
  └─ 注入面板 ────────→ sd-panel → sd-chip → sd-inject
```

**默认参数**：时长15秒 · 画幅9:16 · 风格电影写实 · 无字幕/无BGM

## 子 skill 调用协议

子 skill 不是自动注入的函数。主 skill 在执行每一步前，必须先读取对应目录下的
`SKILL.md`，再把上一步的结果作为下一步输入。不要只根据上面的名称猜测规则。

- 输入：用户原文，以及上一步产出的结构化结果。
- 输出：Markdown 结果，保留角色、场景、台词、动作、时间轴和未解决问题。
- 子 skill 只负责自己的职责，不重复改写上游已经确认的内容。
- `sd-quality` 必须在最终输出前运行一次；发现缺项时先修正，再交付。
- `sd-inject` 属于写入外部面板的操作，必须先展示待修改片段和摘要，得到用户明确确认后才能调用。
- 没有对应子 skill、读取失败或输入不完整时，停止当前步骤并说明缺什么，不要静默跳过。

## 剧本转分段提示词标准流程

### 1. 按集准备输入

从剧本中复制需要处理的集数，默认一集作为一个独立输入文件或一个独立文本块。不要把多集未标记地混在一起，否则台词、场景和资产容易串集。

### 2. 建立场景基础提示词锁定表

分段前先为每个场景建立基础提示词。至少记录：场景名称、地点、时间、内外景、固定风格、固定光线、固定色调、固定空间关系和允许变化的条件。

同一场景的基础提示词默认保持不变。只有日出日落、延时摄影、天气变化、火灾爆炸等剧本明确写出的变化，才允许修改对应字段。

| 场景 | 固定内容 | 允许变化 |
|---|---|---|
| S01 山神庙内景 | 破旧木门、冷蓝夜色、右侧窗户、潮湿石墙 | 火光映红、天亮、明确的烟雾变化 |

### 3. 按分段规则拆分

调用 `sd-segment-split`，按戏剧任务和节拍拆分，不为了凑时长硬切。每段保留所属场景和基础提示词 ID。

### 4. 逐段检查

调用 `sd-quality`，逐段确认：台词与原剧本逐字一致、没有无中生有、台词量能在时长内说完、镜头数量符合段落类型、动作结果连续、基础提示词没有无理由漂移。

### 5. 对应资产

读取工作面板或资产清单，把剧本里的场景名、角色名、道具名映射到真实资产名称和 ID。映射不确定时标记待确认，不能用相似名称或占位符代替。

### 6. 修改阶段安全注入

修改已有工程时，默认不覆盖、不删除原有片段。先由用户或 Agent 在面板中建立新的空白片段，再把修改后的提示词注入空白片段。只有用户明确要求覆盖旧片段时，才允许走单独的覆盖确认流程。

### 7. 使用自然语言描述变化

最终提示词中不要用箭头或符号表达状态变化。使用“随后、紧接着、画面逐渐、人物先……然后……最后……”等自然语言，让模型理解变化过程。箭头只允许出现在内部流程图和人类阅读的结构说明中。
